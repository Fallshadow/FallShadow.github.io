# 1.1 理解 Helper 的定位和加载时机

UWindowTransparencyHelper 是 FWindowTransparencyModule 的一个引用，关卡通过 BPL 激活 Module 获取 Helper。

# 1.2 FWindowTransparencyModule::GetHelper()

插件最主要的功能库 UWindowTransparencyHelper

## 获取插件

FWindowTransparencyModule* Module = FModuleManager::GetModulePtr<FWindowTransparencyModule>("WindowTransparency");

从 UE 的模块管理器里，用名字 "WindowTransparency" 查找你的插件模块。如果找不到，说明插件没被加载，直接返回空。

## 懒加载 HelperInstance 检测

!Module->HelperInstance && !IsRunningCommandlet() && !IsRunningDedicatedServer() && GEngine

防止重复创建 && 当前不是以命令行模式运行 && 当前不是专用服务器 && 全局引擎实例已存在且初始化完毕

### IsRunningCommandlet

过滤无窗口环境。Commandlet 是 UE 的无头批处理模式（比如在服务器上跑资产检查），根本没有游戏窗口，创建 Helper 去拿 HWND 是徒劳的，甚至会崩溃

### IsRunningDedicatedServer

过滤无窗口环境。专用服务器只处理网络逻辑，没有渲染窗口，同样没有 HWND。你的桌宠项目运行时肯定不是 Dedicated Server，这是一层保险

### GEngine

保证游戏视口已就绪。UWindowTransparencyHelper::Initialize() 内部是通过 GEngine->GameViewport->Window 去拿 HWND 的。如果 GEngine 还是空指针，调用 Initialize() 必定失败，Helper 创建出来也没用

## 创建

### NewObject

Module->HelperInstance = NewObject<UWindowTransparencyHelper>();

#### 为什么非要用 NewObject ？

因为 UWindowTransparencyHelper 继承自 UObject，而 UE 引擎里的所有 UObject 及其子类，都禁止使用 C++ 标准的 new 来创建。这不是语法的限制，而是 UE 为了维护它整个对象生命周期管理体系的强制要求。

标准的 C++ new 仅仅是从堆上分配一块原始内存。但 UE 里的 UObject 要想正常工作，必须和引擎的多个核心系统（垃圾回收、反射、序列化）紧密连接。用 NewObject 就像给对象办了全套“户口”，而标准 new 只是一个“黑户”。

具体来说，NewObject 做了这些 new 干不了的事：
- 注册到垃圾回收系统：让GC知道你对象的引用关系，自动管理生命周期。
  - 直接 new 的对象，GC 不认识，可能会被误删，或者导致内存泄漏，你只能手动 delete。
- 初始化反射数据：关联 UClass 元数据，让蓝图、序列化、编辑器能识别和使用你的对象。
  - 对象在蓝图里无法访问，不能保存和加载，只是一个普通的 C++ 对象。
- 自动属性初始化：把对象的所有属性设为安全的默认值。
  - 直接 new 出来的内存是未初始化的，可能带有脏数据，导致崩溃或逻辑错误。


### 防止垃圾回收

Module->HelperInstance->AddToRoot();

AddToRoot()：把这个对象加入 UE 垃圾回收系统的“根节点”。

为什么必须做：UE 有自动垃圾回收（GC）。一个没有被任何根节点或强引用持有的 UObject，随时可能被 GC 清理掉。加上 AddToRoot 等于告诉引擎：“这个对象我手动管理，你千万别帮我删了！” 这样 Helper 单例才能在整个引擎生命周期里一直存活。

#### 难道不这样做真的会回收么？

会，而且非常容易发生。 在 UE 的垃圾回收机制下，如果你不调用 AddToRoot()，你刚创建的 HelperInstance 很可能在下次 GC 周期就被悄无声息地回收掉。

UE 的垃圾回收（GC）不是靠人工 delete，而是基于引用追踪的自动系统。简单说，它的判断标准是："如果一个 UObject 没有任何 UObject 类型的变量引用它，那它就是垃圾，可以被清除。"

谁是引用者？ Module 是引用者，但 Module 本身是一个原生的 C++ 指针 (FWindowTransparencyModule*)。GC 不认原生指针：UE 的 GC 追踪系统只认两种强引用：被 UPROPERTY() 宏标记的成员变量，或被 AddToRoot() 明确挂载到根集上的对象。FWindowTransparencyModule 不是 UObject，它的 HelperInstance 成员只是个原生 C++ 指针，没有 UPROPERTY() 标记。在 GC 眼里，这等于"没有 UObject 在引用这个 Helper"。

结论：在 GC 眼中，这个刚出生的 HelperInstance 处于"没有任何人引用的孤儿状态"。只要内存有点紧张，或者你切换关卡、加载资源，触发了一次常规 GC，这个孤儿就会被直接清除掉。随后，你的代码里留着的那个 Module->HelperInstance 指针，就成了一个悬空的野指针，一调用就崩溃。

### 创建总结

NewObject 负责"出生"：给对象上户口，分配内存，初始化属性。

AddToRoot 负责"保护"：在它的生命周期有明确的管理者接手前，提供一个强力的临时保护，告诉 GC："这家伙我有用，不许动！"。

这个责任分离非常关键。引擎内部创建的大多数 UObject，比如一个 Actor 的组件，会立刻被它的 Owner（一个 UObject）通过 UPROPERTY() 引用，从而获得保护。但你的 HelperInstance 是个由非 UObject 模块持有的全局单例，这种特殊身份，就需要你手动承担起保护它的责任。

所以，AddToRoot 在这里就是 "为非 UObject 所有者提供的、唯一的强引用保护"。它确保对象在模块的整个生命周期内，都牢牢地"钉"在内存里，不会被 GC 的自动清理波及。

# 1.3 初始化

## 快速成功检查

if (bIsInitialized && GameHWnd && IsWindow(GameHWnd) && !bIsDesktopBackgroundActive)

### IsWindow(GameHWnd)

这个句柄是否依然指向一个活着的窗口。GameHWnd 不为空不代表窗口还活着。比如用户关闭了游戏窗口但引擎还没完全退出，那个 HWND 就变成了"死句柄"。IsWindow() 是系统级的存活确认

### bIsDesktopBackgroundActive

自定义的桌面背景模式

## 拿到通用窗口

尝试获取并缓存 UE 引擎的通用窗口引用

if (GEngine && GEngine->GameViewport && GEngine->GameViewport->GetWindow().IsValid())
    GameSWindowPtr = GEngine->GameViewport->GetWindow();

GameSWindowPtr ： TSharedPtr<SWindow>，也就是 Slate 框架里的通用窗口对象。SWindow 是 Slate 框架里的一个“通用窗口”或“平台无关的窗口抽象”。

它的核心作用就是：在所有平台（Windows、Mac、Linux）上，为 UE 提供一个统一的操作窗口的接口，而不用直接跟各个操作系统的原生窗口 API 打交道。

| 平台      | 原生窗口句柄     | 原生窗口系统 |
| ------    | ----------        | - |
| Windows   | HWND	                                | Win32 API |
| macOS	    | NSWindow*	                            | Cocoa / AppKit |
| Linux	    | Window (X11) 或 wl_surface (Wayland)  | X11 / Wayland |

如果引擎直接操作 HWND，那代码就没法在 Mac 上跑。SWindow 就是来解决这个问题的。

# 1.4 GetGameHWnd

## 策略一：从缓存里取（最快路径）

```Cpp
TWeakPtr<SWindow> GameSWindowPtr;

if (GameSWindowPtr.IsValid()) {
    TSharedPtr<SWindow> StrongSWindow = GameSWindowPtr.Pin();
    if (StrongSWindow.IsValid() && StrongSWindow->GetNativeWindow().IsValid()) {
        void* Handle = StrongSWindow->GetNativeWindow()->GetOSWindowHandle();
        UE_LOG(LogWindowHelper, Verbose, TEXT("GetGameHWnd: Returning HWND from cached GameSWindowPtr: %p"), Handle);
        return static_cast<HWND>(Handle);
    }
}
```

这个最快路径的本质是：优先使用缓存窗口对象，但只在不增加风险的前提下。它通过 TWeakPtr + Pin() 的组合，实现了“平时零开销旁观，用时零开销验证”的安全访问模式。

### 为什么要用 TWeakPtr 存着，而不是直接存 TSharedPtr？

- TSharedPtr（强引用）：会强制窗口对象活着。就算系统想销毁它，只要你还拿着强引用，它就死不了。
- TWeakPtr（弱引用）：只是远远地看着，不妨碍窗口销毁。UE 的 Slate 系统可以在不通知你的情况下重建或关闭窗口。

好处：用 TWeakPtr 存着，你不会意外地“卡住”一个本该销毁的窗口，也不会导致内存泄漏。

### Pin() 这个临时升级，为什么是“零开销”？

- Pin() 内部会尝试把弱引用提升为强引用，但它只是临时创建一个局部的 TSharedPtr，用- 完后立刻析构。
- 这个过程没有内存拷贝，只是在引用计数上短暂地加 1，然后立即减 1，几乎可以忽略不计。

它巧妙地解决了“既要旁观，又要安全”的矛盾：平时不妨碍销毁，要用时临时确认你还活着。

## 策略二：从引擎主路径取（常规路径）

```Cpp
if (GEngine && GEngine->GameViewport && GEngine->GameViewport->GetWindow().IsValid())
{
    TSharedPtr<SWindow> GameSWindow = GEngine->GameViewport->GetWindow();
    const_cast<UWindowTransparencyHelper*>(this)->GameSWindowPtr = GameSWindow;
    if (GameSWindow.IsValid() && GameSWindow->GetNativeWindow().IsValid())
    {
        void* Handle = GameSWindow->GetNativeWindow()->GetOSWindowHandle();
        return static_cast<HWND>(Handle);
    }
}
```

缓存失效了（窗口可能被重建过），直接从 GEngine 顶层重新获取。

这里的 const_cast 临时去掉 const 限制，把新窗口存起来

## 策略三：从 Slate 应用层取（兜底路径）

```Cpp
if (FSlateApplication::IsInitialized())
{
    TSharedPtr<SWindow> ActiveWindow = FSlateApplication::Get().GetActiveTopLevelWindow();
    if (ActiveWindow.IsValid() && ActiveWindow->GetNativeWindow().IsValid())
    {
        void* Handle = ActiveWindow->GetNativeWindow()->GetOSWindowHandle();
        return static_cast<HWND>(Handle);
    }
}
```

前两步都失败时，直接去问 Slate 应用层：“现在最顶层、正在用的那个窗口是谁？”

## 为什么需要这么多策略？

因为你的桌宠插件需要应对多种运行场景：
- 独立游戏模式
- 编辑器运行模式
- 窗口被最小化后恢复
- 全屏模式切换

这三种策略从缓存到引擎顶层，再到 UI 框架层，形成一张严密的安全网，确保了在所有可能的启动顺序和窗口状态下，都能成功拿到那个“门牌号”。

# 1.5 设置样式

备份原始窗口样式，主要有基础样式和拓展样式

## GetWindowLongPtr

### GWL_STYLE

LONG_PTR TrueOriginalWindowStyle = GetWindowLongPtr(GameHWnd, GWL_STYLE);

作用：读取窗口最原生的样式

LONG_PTR 一个兼容 32/64 位的整数。把所有样式按二进制位拼在一起，变成一个整数存起来。

### GWL_EXSTYLE

LONG_PTR TrueOriginalExWindowStyle = GetWindowLongPtr(GameHWnd, GWL_EXSTYLE);

保存窗口原始扩展样式

## GetParent

TrueOriginalParentHwnd = ::GetParent(GameHWnd);

保存窗口原始父窗口
- 窗口在 Windows 里是有父子关系的
- UE 游戏窗口刚创建时，一定有一个默认父窗口
- 这里存的是：最开始的亲爹
- 后面你如果把窗口挂到别的窗口、桌面、甚至变成桌面壁纸，都能靠这个恢复回去
