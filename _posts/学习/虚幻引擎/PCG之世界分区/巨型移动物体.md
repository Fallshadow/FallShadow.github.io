针对大规模动态移动的Actor（如巨型飞船、移动平台），社区中探索出的主流方案是 Sub-World Partition + Level Instance（关卡实例） 。这套方案的思路类似于 “用一个小世界，扛着另一个更小的世界移动” 

这套方案的核心是利用 UE5 的 Level Instance 功能，将移动物体及其所有零件打包成一个独立的子关卡。通过为这个子关卡启用 Standalone（独立） 行为，它会拥有一套完全独立的子世界分区（Sub-World Partition） 系统

这样做的好处是，当飞船本体在世界中移动时，其内部零件（如座椅、装饰物）相对于飞船本身是静止的。这套独立的子分区系统可以专注于管理这些内部零件的加载与卸载，而飞船整体在世界坐标系中的移动，则需要通过更底层的手段来实现。

# 具体操作

- 资产准备：在编辑器中将移动物体的所有零件Actor选中，右键选择 “Create Level Instance” 将它们打包。在关卡实例的细节面板中，务必将 Level Behavior 设置为 Standalone，这是激活子世界分区的关键选项
- 处理移动同步：这是最大的技术难点。关卡实例本身被加载后，其内部的Actor默认是静止的。为了实现内部零件跟随飞船骨骼或本体移动，你需要在运行时生成一个“枢轴点Actor”来接管内部所有零件，并通过监听流送状态，在零件被流送加载时，动态地将它们附着（Attach）到正确的目标上

# 难点

涉及引擎源码修改：由于运行时动态更新子世界分区的 InstanceTransform 涉及引擎的私有或编辑器限定 API，你可能必须修改 Unreal Engine 的 C++ 源码才能让这套机制完整运作。一位开发者反馈：“我们尝试将骨骼的世界变换转发给 UWorldPartition::InstanceTransform 来欺骗 WP 的流送源计算……但公开的 setter 仅在编辑器下可用。”

处理异步加载的复杂性：流式加载是异步的。当飞船移动并触发新地块加载时，你无法立即附着新加载的 Actor。你需要通过监听 FLevelStreamingDelegates::OnLevelStreamingStateChanged 等委托来等待加载完成，但社区反馈目前很难在运行时准确识别某个流送关卡属于哪个关卡实例。
